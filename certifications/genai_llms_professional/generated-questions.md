- C. Disable the KV cache entirely for shorter sequences and recompute attention during decoding to free GPU memory
- D. Move the model to 8×H100 GPUs with tensor parallelism set to 8 to distribute the per-GPU compute load
- Answer: A

### Q2: A team trains a 70B model with tensor parallelism across 4 GPUs within a single node. They observe that increasing to 8 GPUs yields only 18% throughput improvement instead of the expected ~90%. Nsight Systems traces show GPU kernels regularly idle waiting on NCCL all-reduce between tensor-parallel ranks. The GPUs are H100s connected via NVLink with NVSwitch. What should the team investigate first?
- ID: gpu-015
- Domain: GPU Acceleration and Optimization
- Section: GPU Acceleration and Optimization
- Topic: distributed training parallelism
- Exam: NCP-GENL
- Difficulty: hard
- A. The communication-computation overlap: NCCL all-reduce may not be overlapping with the next compute kernel, causing serialized execution
- B. The NVLink bandwidth being insufficient for 8-way tensor parallelism on H100 hardware
- C. The model being too small — tensor parallelism at scale 8 only benefits models above 175B parameters
- D. The NCCL algorithm auto-selecting Ring instead of Tree for the all-reduce collective on this GPU count
- Answer: A
- Explanation: Tensor parallelism inserts an all-reduce after every forward and backward pass through a partitioned weight matrix. If NCCL communication is not overlapped with the immediately following GEMM kernel, the GPU stalls waiting for the reduction to complete. This serialization is the most common cause of poor scaling when increasing tensor-parallel ranks — not raw bandwidth limits. NVSwitch on H100 provides 900 GB/s per GPU, which is sufficient; the issue is scheduling, not capacity.
- Why B is wrong: NVSwitch on H100 provides 900 GB/s bidirectional per GPU, which is sufficient for 8-way tensor parallelism on a 70B model. Bandwidth is rarely the bottleneck when GPUs are connected via NVSwitch — scheduling is.
- Why C is wrong: 70B parameters is more than large enough for tensor parallelism to matter. The weight matrices are large enough that splitting them across 4 or 8 GPUs meaningfully reduces per-GPU memory and compute. The poor scaling is an overlap issue, not a model-size threshold.
- Why D is wrong: On NVSwitch-connected GPUs within a single node, NCCL uses NVLS (NVLink SHARP) for all-reduce, not Ring or Tree. Even if it selected the wrong algorithm, the 18% scaling suggests a scheduling/overlap problem, not an algorithm selection one.

- A. Enable activation checkpointing to trade compute for memory so FSDP can use larger gradient accumulation steps, reducing the frequency of all-gather during the backward pass

- Answer: A

- Why D is wrong: Adding GPUs increases the number of ranks participating in NCCL collectives, which increases communication overhead (more participants in all-gather). The root cause is the frequency of communication, not the per-rank shard size.
